# Frontend and Admin API Contract

Base URL for integration:

- Public frontend: `NEXT_PUBLIC_API_BASE_URL` should point to `http://localhost:5000/api/v1` or equivalent.
- Admin app: use `VITE_API_BASE_URL` or equivalent pointing to `http://localhost:5000/api/v1`.

Response envelope:

```json
{
  "success": true,
  "message": "Request completed successfully",
  "data": {},
  "meta": {},
  "requestId": "request-id-from-x-request-id-header",
  "timestamp": "2026-05-09T00:00:00.000Z"
}
```

Error envelope:

```json
{
  "success": false,
  "message": "Error message",
  "data": null,
  "meta": null,
  "errorCode": "VALIDATION_ERROR",
  "requestId": "request-id",
  "errors": [],
  "timestamp": "2026-05-09T00:00:00.000Z"
}
```

`X-Request-Id` is set on every backend response. Error responses include the same value in `requestId`; frontend and admin clients normalize the header value onto their `ApiEnvelope` and `ApiRequestError` objects. Client-side error handling should branch on `status` and `errorCode`, and display `requestId` in support/debug contexts when useful.

## Current Frontend Flow Notes

### NSW Legal Awareness

- Racial abuse and migrant challenge assistant entry screens show NSW legal-awareness pathway cards before the conversation starts.
- `/rag/timeline-assistant` and `/rag/answer` may return `legalAwareness` for NSW racial abuse or migrant challenge contexts. `sourceStatus=approved_sources_used` means citations came from approved, current, legally reviewed sources; `sourceStatus=insufficient_approved_sources` means the UI must show a no-citation, information-only fallback.
- Frontend assistant messages should render backend confidence, disclaimer, human-review state, RAG retrieval availability, citation title/publisher/URL/jurisdiction/update date, and the NSW legal-awareness card when present.
- Detailed legal explanation copy must not be hard-coded in frontend screens; it must come from approved backend RAG sources once the official source registry has approved current material.

### ScamShield Screenshot Upload

- The intake UI has explicit modes for text/SMS paste, URL check, email analysis, and screenshot upload.
- Screenshot upload accepts an image file, shows the selected filename/preview, and waits for explicit `process_with_ai` consent before calling the backend.
- The backend screenshot endpoint accepts multipart field `image` or JSON `imageBase64`; if an image is supplied, OCR is attempted through the configured OpenAI model and the extracted text is scored. If OCR is unavailable, the API returns a clear error and the frontend may show a labelled demo preview instead of presenting a fake result.
- Results should render backend fields when present: `_id`, `riskScore`, `riskLevel`, `confidence`, `indicators`/`redFlags`, `recommendations`, `metadata.extractedTextLength`, and any `extractedEntities`.

### ScamShield Draft And Submit

- The risk page stores the live ScamShield `_id` in session flow state.
- The agency page calls `POST /scamshield/:id/generate-report-draft` and displays the returned `analysis.draftReport`.
- Submit/share calls `POST /scamshield/:id/submit` only after `share_with_agencies` consent is granted and the user has enabled sharing in the UI.
- If the user declines or has not enabled sharing, SafeSpeak keeps the draft prepared/local and must not imply an external agency received the report.

### Report Review And Success

- If a `report_id` exists, review loads `GET /reports/:id`, `GET /reports/:id/status`, and `GET /reports/:id/timeline`.
- Review/success pages show the backend report id/ref and current status. Guidance cards are labelled as guidance only.
- The review continue action saves `status: "ready_for_review"`; it does not claim external submission. Success/share copy should use prepared/saved language unless the backend submission record has `actuallySent=true` or status `submitted`/`acknowledged`. `requires_manual_action` and `config_missing` are auditable attempts, not external sends.
- Report history uses `GET /reports` and may call `GET /reports/:id/status` for status labels. Static/mock history should be hidden or explicitly labelled as demo only.

### Support Fallback

- Explorer prefers `GET /support/services` and `POST /support/recommendations`.
- Static resource cards appear only when backend services are unavailable or empty, and they are labelled "Local fallback resources" or "Example resources".
- Service details prefer `GET /support/services/:id`; local fallback detail copy is labelled when backend detail loading fails.

### Consent Enforcement

- Protected actions call `GET /consents/current` through `ensureConsent` before the protected request.
- `POST /consents/update` is only called from explicit user actions such as ConsentRequiredCard "Allow and continue" or the settings consent center.
- "Not now" dismisses the prompt and does not call the blocked backend action.

### RAG Readiness Notes

- Internal SafeSpeak rule documents belong in `knowledge/internal/` and ingest as `internal_product_rule`.
- Official legal/support sources ingest from approved official government, agency, or AustLII domains only.
- RAG is the AI knowledge mode for legal/source content; do not fine-tune model weights on statutes, user reports, chats, evidence, case stories, confidential advice, third-party commentary, or private/internal legal material.
- Every official knowledge source must carry jurisdiction, topic, source type, publisher, licence status, source URL, version/update metadata, and review schedule metadata.
- Official legal sources remain `pending_review` with `legalReviewed=false` by default.
- Public legal RAG answers use only `approved` official legal sources with `legalReviewed=true` and a future `nextRefreshAt` date.
- Updating material governance fields on an approved source returns it to `pending_review` until another admin approves it again.
- Official URL refresh keeps unchanged approved material current, but changed or metadata-only official material returns to review and is excluded from RAG until approved again.
- If no approved authoritative legal source exists, `/rag/answer` must return a safe insufficient-sources fallback without invented citations.
- RAG citations include title, publisher, URL, jurisdiction, source type/category/topic, and update date.
- Crisis and core SafeSpeak behavior fallbacks may come from hard guardrails even when vector retrieval is unavailable.
